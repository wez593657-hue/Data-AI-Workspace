# 15 人工审批流程

> 历史参考：本文件保留旧版人工审批流程。当前不经过 PR 或人工代码审查，用户确认脚本和校验结果后直接在 `master` 分支提交和推送；实际执行规则以 `CONTRIBUTING.md` 为准。

## 1. 审批体系概述

### 1.1 审批目的

人工审批是开发流程的最后一道防线，确保所有变更符合业务需求、技术规范和安全合规要求。

### 1.2 审批层级

| 层级 | 审批人 | 审批内容 | 通过条件 |
|------|--------|----------|----------|
| L1 | 技术负责人 | 技术方案、代码质量 | 代码审查通过 |
| L2 | 业务负责人 | 业务需求、数据准确性 | 符合业务需求 |
| L3 | 安全合规负责人 | 数据安全、合规性 | 符合安全规范 |

### 1.3 审批流程

```
开发完成 → 自动校验 → 代码审查 → L1审批 → L2审批 → L3审批 → 合并发布
```

## 2. 审批范围

### 2.1 需要审批的变更

- [ ] 新增表结构
- [ ] 修改表结构（含字段增删改）
- [ ] 数据字典变更
- [ ] Mapping映射变更
- [ ] ETL脚本变更
- [ ] 存储过程变更
- [ ] 文档重大变更

### 2.2 无需审批的变更

- [ ] 代码格式调整
- [ ] 文档错别字修正
- [ ] 注释完善
- [ ] 测试数据更新
- [ ] 工具脚本优化（不影响业务逻辑）

## 3. 审批流程详解

### 3.1 L1 技术审批

**审批人**: 技术负责人

**审批内容**:

- 技术方案合理性
- 代码质量和规范性
- 架构设计合理性
- 性能影响评估
- 可维护性评估

**审批标准**:

| 检查项 | 通过标准 |
|--------|----------|
| 代码规范 | 符合项目SQL规范 |
| 设计文档 | 完整且通过评审 |
| 自动校验 | 全部通过 |
| 代码审查 | 无阻塞问题 |
| 性能 | 满足性能要求 |

**审批时限**: 2个工作日

### 3.2 L2 业务审批

**审批人**: 业务负责人

**审批内容**:

- 业务需求符合度
- 数据准确性
- 业务流程正确性
- 业务规则一致性

**审批标准**:

| 检查项 | 通过标准 |
|--------|----------|
| 需求文档 | 已确认需求范围 |
| 数据模型 | 符合业务需求 |
| 测试结果 | 通过业务测试 |
| 验收标准 | 满足验收要求 |

**审批时限**: 3个工作日

### 3.3 L3 安全合规审批

**审批人**: 安全合规负责人

**审批内容**:

- 数据安全
- 隐私合规
- 权限控制
- 监管要求

**审批标准**:

| 检查项 | 通过标准 |
|--------|----------|
| 敏感数据 | 已脱敏处理 |
| 数据加密 | 符合加密要求 |
| 权限控制 | 最小权限原则 |
| 合规性 | 符合行业规范 |

**审批时限**: 2个工作日

## 4. 审批流程

### 4.1 提交审批

1. 开发者完成开发并通过自动校验
2. 创建Pull Request
3. 填写完整的PR模板
4. 提交审批请求

### 4.2 审批执行

1. CODEOWNERS自动分配审批人
2. 审批人按层级顺序审批
3. 每个审批人独立审查
4. 审批人可提出修改意见

### 4.3 审批结果

| 结果 | 处理方式 |
|------|----------|
| 通过 | 进入下一审批层级 |
| 有条件通过 | 开发者修改后重新提交 |
| 不通过 | 开发者重新设计，重新提交 |

### 4.4 审批记录

所有审批记录应包含：

- 审批人
- 审批时间
- 审批意见
- 审批结果

## 5. 审批权限

### 5.1 角色权限

| 角色 | L1审批 | L2审批 | L3审批 | 合并权限 |
|------|--------|--------|--------|----------|
| 技术负责人 | ✓ | - | - | ✓ |
| 业务负责人 | - | ✓ | - | - |
| 安全合规负责人 | - | - | ✓ | - |
| 项目负责人 | ✓ | ✓ | ✓ | ✓ |

### 5.2 权限管理

- 权限由CODEOWNERS文件定义
- 权限变更需项目负责人批准
- 审批人需具备相应专业能力

## 6. 审批例外

### 6.1 紧急变更

紧急变更可跳过部分审批流程，但需满足：

- 有明确的紧急原因
- 已通知所有审批人
- 事后补全审批记录
- 项目负责人批准

### 6.2 审批超时

若审批超时（超过审批时限）：

- 自动通知审批人
- 自动升级到上级审批人
- 记录超时原因

### 7.2 审批流程管理

- Issue追踪系统
- 审批状态看板
- 审批记录归档

## 8. 审批监督

### 8.1 审批质量监控

- 审批通过率统计
- 审批时限监控
- 审批意见分析

### 8.2 审计日志

- 所有审批操作记录
- 审批变更历史
- 审计报告生成

## 9. 责任界定

### 9.1 审批人责任

- 审批人对审批结果负责
- 审批人需充分审查
- 审批人需记录审批意见

### 9.2 开发者责任

- 开发者需确保代码质量
- 开发者需提供完整文档
- 开发者需响应审批意见

### 9.3 项目负责人责任

- 项目负责人对整体流程负责
- 项目负责人需协调审批冲突
- 项目负责人需处理审批例外

## 10. 流程改进

### 10.1 定期回顾

- 每季度回顾审批流程
- 收集审批人反馈
- 优化审批流程

### 10.2 持续改进

- 根据反馈调整审批标准
- 优化审批时限
- 改进审批工具
