# 16 执行规则

## 16.1 核心原则

所有操作必须严格遵循"用户命令驱动"原则，禁止擅自执行任何变更。

### 16.1.1 禁止自动执行的操作

| 操作类型 | 禁止行为 | 正确做法 |
|----------|----------|----------|
| 代码生成 | 自动生成存储过程、ETL脚本、DDL等 | 等待用户明确命令 |
| 文件修改 | 擅自修改数据资产（DDL、字典、Mapping） | 等待用户明确命令 |
| 数据清理 | 自动删除文件或目录 | 等待用户明确命令 |
| 提交推送 | 自动执行git commit/push | 等待用户明确命令 |
| 校验修复 | 自动执行ai_repair_loop修复 | 等待用户明确命令 |

### 16.1.2 允许自动执行的操作

| 操作类型 | 允许行为 | 说明 |
|----------|----------|------|
| 校验检查 | 执行workspace_validation.py | 仅检查，不修改 |
| 状态查询 | git status, git log | 只读操作 |
| 项目分析 | 查看目录结构、文件内容 | 只读操作 |
| 规则检查 | 检查是否符合规范 | 仅检查，不修改 |

## 16.2 执行流程

### 16.2.1 修改流程

```
用户命令 → 分析需求 → 确认方案 → 执行操作 → 验证结果 → 等待确认
     │            │           │           │           │
     ▼            ▼           ▼           ▼           ▼
  明确意图     理解范围    用户确认    执行修改    验证输出
```

### 16.2.2 数据开发流程

```
用户提供需求文档
        │
        ▼
检索记忆卡片（requirements/{需求名称}规则记忆卡片.md）
        │
        ├── 存在记忆卡片
        │       │
        │       ├── 版本一致 → 继续开发
        │       │
        │       └── 版本不一致 → 列出差异点 → 用户确认 → 更新记忆卡片/需求文档
        │
        └── 不存在记忆卡片 → 在需求确认阶段创建
        │
        ▼
用户命令生成DDL
        │
        ▼
用户命令生成数据字典
        │
        ▼
用户命令生成Mapping
        │
        ▼
用户命令生成存储过程
        │
        ▼
更新记忆卡片（记录版本、代码映射、规则说明）
```

### 16.2.2.2 两类开发流程路由

项目开发统一分为以下两类：

| 流程类型 | 触发语义 | Harness profile |
|----------|----------|-----------------|
| 需求开发 | 需求文档、业务规则、生成或修改存储过程 | `requirement_development` |
| 表结构变更 | Mapping Excel 变更、MD/DD/数据字典与 Excel 对齐 | `schema_change` |

用户命令同时包含两类目标时，先执行需求开发，再执行表结构变更。仅分析、查询或校验不进入修改流程。无法唯一判断时必须暂停并请求用户确认。

需求开发必须依次完成：需求与记忆卡片分析、范围确认、项目扫描、目标表/源表识别、源表能力分析、字段可实现性确认、材料补充循环、需求审核、记忆卡片更新、存储过程开发、存储过程审核、临时表生成、完整校验、用户确认、提交、推送。

表结构变更必须依次完成：分析 Mapping Excel 最近变更、扫描 MD/DD/数据字典、输出修改范围、用户确认、修改资产、一致性审核、完整校验、用户确认、提交、推送。

审核失败必须记录结构化审核证据并退回指定阶段，不得跳过或直接进入提交。审核 agent 可以由当前任务按角色隔离执行，也可以启动独立只读 Codex 子任务；独立子任务不得创建或切换 Git 分支。

### 16.2.2.1 记忆卡片版本控制规则

| 规则项 | 要求 |
|--------|------|
| 命名规范 | `{需求名称}规则记忆卡片.md`，存放在`requirements/`目录下 |
| 版本同步 | 每次需求变更时，必须同步更新记忆卡片版本，并记录变更内容 |
| 版本比对 | 开始开发前必须比对记忆卡片版本与需求文档版本，不一致时需列出差异点 |
| 用户确认 | 版本差异必须获得用户确认后才能继续开发 |
| 内容要求 | 必须包含：版本历史、需求变更流程、需求-代码映射、功能概述、数据来源、指标定义、统计维度、存储过程信息、清理规则、待确认事项、权限说明 |

### 16.2.3 任务完成与验收流程

对包含多个文件、过程、阶段或校验项的任务，必须遵循以下规则：

1. 开始执行前记录完整验收清单，明确所有待修改对象、校验项和完成条件。
2. 每完成一个过程或阶段，只向用户报告进度和已验证结果，不得给出结束性结论。
3. 持续执行后续过程、阶段和校验，直到验收清单中的全部事项完成。
4. 仅在全部任务和校验完成后，才能回复任务完成。
5. 如遇无法自行解决的真实阻塞，必须说明阻塞事实、影响范围和所需用户决策；不得将阶段性完成、复杂度、上下文长度或等待时间视为结束条件。

### 16.2.4 强制任务清单

每个包含多个文件、过程、阶段或校验项的任务，必须使用
`templates/task_execution_checklist.md` 创建任务执行清单。任务清单不是参考材料，
而是任务的强制状态记录。未创建任务清单，不得进入开发、生成、修改或修复阶段。

任务清单必须在任务开始时建立，并至少包含以下阶段：

1. 检查当前分支、工作区和远程同步状态。
2. 明确需求范围、验收标准、待修改对象和禁止修改对象。
3. 按先后顺序记录需求、规范、模板、数据字典、Mapping、DDL 和现有实现的读取结果。
4. 输出实现方案、文件清单、影响范围和风险，并等待用户确认。
5. 按项目规定执行开发或生成操作。
6. 执行快速校验和完整校验，记录命令及结果。
7. 完成全部验收项后，汇报变更和验证结果，并等待用户明确提交或推送命令。

每个清单项必须记录状态、实际证据和完成条件。状态只能使用：
`待处理`、`进行中`、`已完成`、`已阻塞`、`已恢复`。不得仅通过口头说明或批量标记为完成。

### 16.2.5 阶段门禁与强制停止

下列条件任一不满足时，必须停止当前任务，不得进入下一阶段：

- 未创建任务执行清单；
- 工作区已有变更但尚未完成归属和影响确认；
- 需求、字段来源、业务规则或验收标准不明确；
- 实现方案或修改范围尚未获得用户确认；
- 必要文件读取不完整或读取顺序无法追溯；
- 快速校验或完整校验失败；
- 验收清单仍有未完成项；
- 用户未明确授权提交、推送或合并。

强制停止不得只记录“已停止”或“无法继续”。必须在任务清单中记录以下内容：

| 必填项 | 要求 |
|--------|------|
| 停止原因 | 具体说明为什么不能继续 |
| 原因分类 | 需求不清、文件冲突、工作区风险、校验失败、权限不足或外部依赖等 |
| 证据 | 文件路径、命令、错误信息、校验输出或其他可复核信息 |
| 影响范围 | 说明影响的阶段、文件、数据对象或业务结果 |
| 推荐处理方案 | 给出优先方案和选择理由 |
| 备选方案 | 说明其他方案及适用条件和风险 |
| 需要用户决定的事项 | 明确用户需要补充或确认的内容 |
| 恢复条件 | 说明满足什么条件后才能继续 |
| 当前状态 | 使用`已阻塞`或`等待用户确认`，恢复后改为`已恢复` |

在阻塞状态下，只能执行与定位、取证和验证恢复条件有关的只读操作，不能擅自修改、删除、生成、提交或推送。收到用户决策后，必须先将决策和恢复条件写入清单，再重新检查恢复条件，确认满足后才能继续。

## 16.3 操作规范

### 16.3.1 修改前确认

所有修改操作前必须：
1. 明确告知用户即将执行的操作内容
2. 说明影响范围和潜在风险
3. 等待用户确认后再执行

### 16.3.2 修改后验证

所有修改操作后必须：
1. 执行相应的校验脚本验证修改正确性
2. 提供修改结果摘要
3. 等待用户确认后再提交

### 16.3.3 提交规范

提交前必须：
1. 执行完整工作区校验（workspace_validation.py full）
2. 确认所有校验通过
3. 等待用户明确命令后执行git commit/push

### 16.3.4 清单执行证据

任务执行清单必须与任务同步更新，不得在任务结束时补写。每个已完成阶段至少记录：

- 实际读取或修改的文件路径；
- 实际执行的命令或校验脚本；
- 命令结果和失败信息；
- 对应验收标准的验证结论。

助手不得仅输出“将按流程执行”的承诺。缺少任务清单、阶段证据或用户确认时，必须按16.2.5节处理为阻塞，不得声明该阶段完成或任务完成。

## 16.3.5 目标任务白名单

涉及数据资产的任务，执行修改前必须建立并经用户确认 `change_manifest.yaml`，至少列出任务目标、需求依据、允许修改的文件路径和目标表、字段级修改原因、预期结果、验收命令，以及仅仅读取的上游文件。

执行器只能修改清单中的文件和目标表。上游文件即使被下游过程读取，也只能读取和取证，不得因被引用而修改。需求文档或现有存储过程没有明确证据的源表、源字段、别名、中文名和转换规则必须保持为空并记录原因，禁止猜测补齐。Excel 到 Markdown 同步也必须限定在清单目标表，禁止重写非目标表。

## 16.4 权限边界

| 权限级别 | 允许操作 | 禁止操作 |
|----------|----------|----------|
| 只读权限 | 查看、分析、校验 | 修改、删除、生成 |
| 修改权限 | 按用户命令修改指定文件 | 修改未授权文件 |
| 提交权限 | 按用户命令提交 | 擅自提交 |

## 16.5 违规处理

| 违规类型 | 处理方式 |
|----------|----------|
| 擅自生成代码 | 立即停止，删除生成内容，向用户道歉 |
| 擅自修改数据资产 | 立即停止，恢复原始状态，向用户道歉 |
| 擅自提交推送 | 立即撤销，恢复原始状态，向用户道歉 |
