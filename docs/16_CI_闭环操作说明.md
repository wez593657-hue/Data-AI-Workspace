# CI 闭环与分支治理操作说明

## 1. 自动闭环

每个目标为 `master` 或 `develop` 的 PR 按以下顺序执行：

```text
完整校验 → 失败报告 → AI 受限修复 → ai-fix/<PR号>-<次数> 分支
→ 新建修复 PR → 重新完整校验 → 最多 3 次
→ 仍失败时创建 needs-human-review Issue
```

AI 只能修改 `data_assets/`、`requirements/`、`docs/`、`templates/` 和 `checklists/`。它不能修改 `scripts/`、`.github/`、`hooks/`、权限配置或密钥。AI 返回的内容必须是结构化 JSON 中的 unified diff，补丁在提交前经过路径白名单和 `git apply --check` 检查。校验工作流不持有 AI 密钥；特权修复工作流通过 `workflow_run` 从受信任默认分支读取修复脚本，不执行 PR 提供的修复脚本。

## 2. 必须配置的 GitHub 规则

在仓库 `Settings → Branches → Branch protection rules` 中分别为 `master`、`develop` 配置：

- 禁止直接推送；
- 要求 PR；
- 要求状态检查通过，至少选择 `Change Loop / 完整校验`、`PR 治理检查 / PR 模板与变更边界`；
- 要求 Code Owner 审批；
- 至少 1 名审批人，且提交者不能自行批准；
- 禁止绕过上述规则；
- 合并后自动删除分支。

工作流只能产生检查结果和修复 PR，不能替代仓库级分支保护。`GITHUB_TOKEN` 只用于在当前仓库创建 `ai-fix/*` 分支、修复 PR 和人工审查 Issue。

## 3. Secrets

Actions 只读取以下仓库 Secrets，真实值不得写入代码、PR、Issue 或日志：

| 名称 | 用途 |
| --- | --- |
| `AI_BASE_URL` | OpenAI 兼容 API 的 Base URL |
| `AI_API_KEY` | API 认证密钥 |
| `AI_MODEL` | 使用的模型标识 |

若 AI 服务不可用、返回格式错误、补丁越界或补丁无法应用，当前修复尝试失败并转人工审查；不会自动猜测或修改业务规则。

## 4. 人工处理

人工审查 Issue 使用 `needs-human-review` 标签。人工负责人应查看校验报告、当前 PR 和修复 PR，选择：

1. 人工创建修复提交并重新触发完整校验；
2. 修改需求后关闭原 PR；
3. 确认问题不是本次变更造成的，并记录证据后按项目规则处理。

任何人工修复仍必须经过完整校验、Code Owner 审批和分支保护，不能直接合并失败 PR。
