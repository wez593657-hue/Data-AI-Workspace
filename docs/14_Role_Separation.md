# AI Agent 角色分工规范

> 历史参考：本文件保留旧版开发、审查和审批角色设计。当前不设置 PR 或代码审查门禁，用户确认脚本后直接在 `master` 分支提交和推送；实际执行规则以 `CONTRIBUTING.md` 为准。

## 1. 角色定义

### 1.1 开发Agent

| 属性 | 内容 |
|------|------|
| **职责** | 代码实现、需求分析、设计文档编写 |
| **可读目录** | 所有目录 |
| **可写目录** | `data_assets/`, `requirements/`, `docs/`, `templates/`, `checklists/` |
| **禁止目录** | `.github/workflows/`, `scripts/`, `.git/hooks/` |
| **禁止操作** | 执行校验脚本、修改校验脚本、提交代码、跳过审批 |

### 1.2 校验Agent

| 属性 | 内容 |
|------|------|
| **职责** | 执行自动校验、生成校验报告、检测问题 |
| **可读目录** | 所有目录 |
| **可写目录** | 无（只读模式） |
| **禁止操作** | 修改业务代码、修改校验脚本、提交代码、批准代码 |

### 1.3 审查Agent

| 属性 | 内容 |
|------|------|
| **职责** | 代码审查、提出修改建议、验证规范遵循 |
| **可读目录** | 所有目录 |
| **可写目录** | 无（只读模式） |
| **禁止操作** | 直接修改业务代码、修改校验脚本、批准代码 |

### 1.4 审批Agent

| 属性 | 内容 |
|------|------|
| **职责** | 最终审批、合并代码、发布版本 |
| **可读目录** | 所有目录 |
| **可写目录** | 无（仅批准操作） |
| **禁止操作** | 参与开发、修改业务代码、修改校验脚本 |

## 2. 角色交互流程

### 2.1 标准开发流程

```
用户需求 → 开发Agent → 代码实现 → 校验Agent → 校验报告 → 审查Agent → 审查意见 → 开发Agent → 修改代码 → 校验Agent → 审查Agent → 审批Agent → 合并发布
```

### 2.2 流程节点说明

| 节点 | 执行Agent | 输入 | 输出 | 失败处理 |
|------|-----------|------|------|----------|
| 需求分析 | 开发Agent | 用户需求 | 需求文档 | 重新分析 |
| 代码实现 | 开发Agent | 需求文档 | 代码文件 | 重新实现 |
| 自动校验 | 校验Agent | 代码文件 | 校验报告 | 开发Agent修改 |
| 代码审查 | 审查Agent | 代码文件+校验报告 | 审查意见 | 开发Agent修改 |
| 最终审批 | 审批Agent | 代码文件+校验报告+审查意见 | 批准/拒绝 | 退回修改 |

| `requirements/` | ✅ 可写 | ❌ 只读 | ❌ 只读 | ❌ 只读 |
| `docs/` | ✅ 可写 | ❌ 只读 | ❌ 只读 | ❌ 只读 |
| `scripts/` | ❌ 禁止 | ❌ 只读 | ❌ 只读 | ❌ 只读 |

| 执行校验脚本 | ❌ | ✅ | ❌ | ❌ |
| 生成校验报告 | ❌ | ✅ | ❌ | ❌ |
| 代码审查 | ❌ | ❌ | ✅ | ❌ |
| 提出审查意见 | ❌ | ❌ | ✅ | ❌ |
| 批准代码 | ❌ | ❌ | ❌ | ✅ |
| 合并代码 | ❌ | ❌ | ❌ | ✅ |
| 提交代码 | ❌ | ❌ | ❌ | ✅ |

### 3.3 流程边界

| 规则 | 说明 | 违反处理 |
|------|------|----------|
| 必须走流程 | 所有代码变更必须经过校验→审查→审批 | Git钩子/CI阻止 |
| 禁止跳过校验 | 代码提交前必须通过自动校验 | Git钩子阻止 |
| 禁止绕过审查 | PR必须经过CODEOWNERS审查 | CI阻止合并 |
| 禁止直接推送保护分支 | 必须通过PR合并 | Git钩子阻止 |

## 4. 角色转换规则

### 4.1 禁止转换

| 当前角色 | 禁止转换为 | 原因 |
|----------|-----------|------|
| 开发Agent | 校验Agent | 避免自我校验 |
| 开发Agent | 审查Agent | 避免自我审查 |
| 开发Agent | 审批Agent | 避免自我审批 |
| 校验Agent | 开发Agent | 避免校验后修改 |
| 审查Agent | 开发Agent | 避免审查后修改 |
| 审查Agent | 审批Agent | 避免审查与审批合并 |

### 4.2 允许转换

| 当前角色 | 允许转换为 | 条件 |
|----------|-----------|------|
| 审批Agent | 开发Agent | 新建需求 |
| 审批Agent | 审查Agent | 紧急情况下 |

## 5. 紧急情况处理

### 5.1 紧急修复流程

```
紧急问题 → 开发Agent → 代码修复 → 校验Agent → 校验通过 → 审批Agent → 紧急审批 → 合并发布
```

### 5.2 紧急修复条件

| 条件 | 说明 |
|------|------|
| 生产事故 | 影响生产环境运行 |
| 数据错误 | 导致数据不一致 |
| 安全漏洞 | 存在安全风险 |

### 5.3 紧急修复记录

所有紧急修复必须记录到Issue中：
- 问题描述
- 修复方案
- 审批记录
- 回滚方案

## 6. 角色协作示例

### 6.1 到期承接需求

```
用户: 开发到期承接功能
    ↓
开发Agent: 创建分支 feature/maturity-takeover
开发Agent: 实现DWD/DWS/ADS层表结构
开发Agent: 提交代码（触发Git钩子校验）
    ↓
校验Agent: 执行 validate_consistency.py
校验Agent: 执行 validate_generated_files.py
校验Agent: 生成校验报告（通过）
    ↓
审查Agent: 审查代码
审查Agent: 提出修改建议
    ↓
开发Agent: 修改代码
开发Agent: 重新提交（再次触发校验）
    ↓
审查Agent: 批准代码
    ↓
审批Agent: 合并到develop
```

### 6.2 Bug修复

```
用户: 修复DDL字段类型错误
    ↓
开发Agent: 创建分支 bugfix/ddl-type-error
开发Agent: 修复字段类型
开发Agent: 提交代码（触发Git钩子校验）
    ↓
校验Agent: 执行校验（通过）
    ↓
审查Agent: 审查代码（通过）
    ↓
审批Agent: 合并到develop
```

## 8. 监督机制

### 8.1 定期审计

| 审计项 | 频率 | 负责人 |
|--------|------|--------|
| 提交记录审计 | 每周 | 审批Agent |
| 校验通过率统计 | 每周 | 校验Agent |
| 审查质量评估 | 每月 | 审批Agent |
| 角色权限检查 | 每月 | 审批Agent |

### 8.2 违规处理

| 违规类型 | 处理方式 |
|----------|----------|
| 跳过校验 | 撤销提交，重新执行 |
| 绕过审查 | 撤销合并，重新PR |
| 直接推送保护分支 | 撤销推送，重新PR |
| 越权操作 | 撤销变更，重新分配角色 |

## 9. 总结

### 9.1 核心原则

1. **角色分离**：开发、校验、审查、审批独立执行
2. **只读约束**：校验Agent和审查Agent只能读取，不得修改
3. **流程强制**：通过技术手段强制走完整流程
4. **责任追溯**：每个操作都有明确的责任人

### 9.2 流程保障

| 层级 | 保障方式 |
|------|----------|
| 本地开发 | Git钩子强制校验 |
| 分支推送 | Git钩子分支检查 |
| PR提交 | CI流水线自动校验 |
| 代码审查 | CODEOWNERS强制审查 |
| 合并发布 | 审批Agent最终批准 |
