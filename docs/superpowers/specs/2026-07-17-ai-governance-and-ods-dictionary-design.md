# AI 治理与 ODS 数据字典补齐设计

## 目标

建立一套不限制 AI 在特性分支生成 SQL、ETL、存储过程和 TMP_ 物理临时表的治理机制。受保护分支的合并和生产发布必须通过自动校验与人工审核。

同时，根据 `data_assets/ddl/ods/` 中的 ODS DDL 补齐 ODS 数据字典。DDL 是结构事实来源；无法从 DDL 推导的业务信息必须显式标记为 `【待确认】`。

## 范围

本次实施包括：

- 修复 Git hooks、安装脚本和 GitHub Actions 的已知运行问题。
- 生成并维护 ODS 数据字典，输出解析异常报告。
- 为存储过程的 TMP_ 物理临时表增加可审核的资产清单和一致性校验。
- 更新模板、检查清单和治理文档，使其与实际执行机制一致。

本次不生成不存在业务需求支撑的存储过程，也不编造字段中文含义、枚举、负责人或业务规则。

## ODS 数据字典

新增 ODS DDL 解析工具，递归读取所有 `.sql` 文件，并为每张可识别的表生成 `data_assets/data_dictionary/ods/<表名>_dd.md`。

解析器提取：表名、字段名、数据类型、长度或精度、可空性、默认值、主键、唯一约束，以及 DDL 注释中的来源信息。输出中未能从 DDL 得出的中文名称、字段中文说明、枚举说明和负责人统一填充 `【待确认】`。

不支持或解析失败的 DDL 不会被静默忽略。工具写入结构化异常报告，列出源文件、失败原因和建议人工处理方式；对应文件不会生成不可信数据字典。

ODS 一致性校验覆盖 DDL 与数据字典的文件配对，并对已生成字典执行字段级比较。现有 DWD、DWS、ADS 校验也升级为字段级比较。

## TMP_ 临时表治理

存储过程可以在特性分支生成物理表 `TMP_`。每张 TMP_ 表必须有一份同名审核清单，包含：

- 表名、所属过程、用途和数据来源。
- 字段名、类型、长度、可空性、默认值和索引。
- 生命周期、清理方式、失败补偿和并发隔离策略。
- 审核状态与审批记录。

校验器从存储过程和 TMP_ DDL 中识别表及字段，并校验三者一致：过程引用、DDL、审核清单。生成阶段只输出问题；PR 合并阶段对受保护分支要求清单完整、字段一致且由 Code Owner 审核。

没有具体需求时，`data_assets/stored_procedure/` 可以只有目录说明和模板，不要求预置过程脚本。

## 自动化与钩子

Git hooks 源文件保存在 `hooks/`，安装程序必须在任一安装失败时返回非零，并支持验证已安装版本是否与源文件一致。

提交信息校验迁移到 `commit-msg`，避免 `prepare-commit-msg` 在编辑前运行造成的绕过。脚本输出使用当前控制台可编码的文本或 UTF-8 安全输出。

本地校验只处理暂存文件；完整校验由 GitHub Actions 执行。工作流统一使用仓库相对路径，Review 检查调用正确的 GitHub API，并由 GitHub 分支保护承担 Code Owner 审批与必需状态检查。

## 校验级别

| 场景 | 行为 |
|------|------|
| AI 本地生成 | 允许生成，输出待确认项和解析问题 |
| 特性分支提交 | 执行快速结构校验，不阻止未审核 TMP_ 草稿 |
| 向 develop/master 的 PR | 执行完整校验；TMP_ 清单、DDL、字段和人工审核缺失时失败 |
| 生产发布 | 仅使用已合并且已审批的版本 |

## 风险与测试

DDL 方言和历史格式可能导致部分 ODS 文件无法自动解析；这些文件必须出现在异常报告中，人工处理后再纳入完整性校验。

测试覆盖解析器的标准 DDL、带 schema 的 DDL、约束与注释、无法识别 DDL；并覆盖 TMP_ 清单缺失、字段不一致、未审核、审核通过等状态。GitHub Actions 至少验证 Windows 无关路径、非零退出码和 PR 事件行为。
