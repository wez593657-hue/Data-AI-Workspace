# 阶段 5：DDL、数据字典和 Mapping 一致性门禁设计

## 1. 目标

对当前仓库全部数据资产执行 DDL、数据字典和 Mapping 的字段级一致性基线扫描，并将结果纳入 Harness 阶段门禁。任何无法由明确文件关系证明的一致性问题都必须报告并阻塞，不自动猜测、替代或补齐业务字段。

## 2. 范围

扫描范围包括：

- `data_assets/ddl/` 下的 ODS、DWD、DWS、ADS DDL；
- `data_assets/data_dictionary/` 下对应层级的数据字典；
- `data_assets/mapping/` 下 ODS 到 DWD、DWD 到 DWS、DWS 到 ADS Mapping；
- 当前 Harness 任务允许的报告、证据和血缘产物路径。

本阶段不修改业务数据资产，不生成或修正字段定义。用户明确下达补齐命令后，才允许在任务白名单内修改指定文件，并重新执行全量扫描。

## 3. 校验模型

统一使用“层级 + 规范化表名 + 规范化字段名”作为字段匹配键，并保留原始文件路径和原始名称用于报告。校验内容包括：

1. DDL 与数据字典的表、字段、类型、长度、精度、可空性、主键和注释；
2. Mapping 目标字段是否存在于目标层数据字典；
3. Mapping 来源表和来源字段是否有明确声明；
4. Mapping 转换规则是否为空或未解析；
5. 同一目标字段存在多个冲突来源时是否已明确确认。

无法解析的 SQL、非标准字典格式或不明确的 Mapping 关系统一标记为 `UNRESOLVED`，不得降级为通过。

## 4. 产物和证据

每个 Harness 任务生成：

- `.harness/tasks/<task-id>/reports/schema-consistency.yaml`：结构化差异报告；
- `.harness/tasks/<task-id>/reports/artifact-graph.yaml`：明确 Mapping 关系形成的血缘图；
- 对报告输入文件记录 SHA-256、读取顺序和校验结果证据；
- 对每个阻塞问题写入 `blocking.yaml`，包含原因、证据、影响、建议、备选方案、用户决策和恢复条件。

报告至少包含 `passed`、`blocked`、`unresolved`、`missing`、`extra`、`mismatch` 和 `conflict` 分类，并逐项引用源文件。

## 5. 门禁和授权

阶段校验通过的条件：

- 所有声明的输入文件均存在且哈希稳定；
- DDL、数据字典和 Mapping 均完成解析；
- 不存在未处理的差异、冲突或 `UNRESOLVED` 项；
- 报告、血缘图和证据记录完整。

发现不一致时，任务进入阻塞路径。恢复必须同时满足：

- 用户明确指定补齐范围和目标文件；
- 修改路径通过 Harness 白名单；
- 修改后重新扫描的报告通过；
- 恢复证据引用新的文件哈希和验证结果。

补齐授权与发布授权分离。补齐不等于允许提交或推送。

## 6. 实现边界

- 新增独立 `scripts/harness/schema_consistency.py`，复用现有基础解析和证据接口；
- 新增或扩展 `scripts/harness/artifact_graph.py`；
- 通过 CLI 和 `gate_checker.py` 接入阶段门禁；
- 不把业务字段修复逻辑写入扫描器；
- 不自动覆盖用户已有修改；
- 不改变现有业务 SQL、DDL、数据字典和 Mapping 内容。

## 7. 测试和验收

测试至少覆盖：

- 全量基线通过；
- 字段缺失和多余字段；
- 类型、长度、精度、可空性和注释差异；
- Mapping 目标字段缺失；
- 来源字段缺失或转换规则为空；
- 多来源冲突；
- 无法解析项阻塞；
- 报告和证据哈希变化后失效；
- 未授权补齐被拒绝；
- 授权补齐后重新扫描通过；
- 阶段门禁不能被不完整报告绕过。

验收结果必须同时说明执行逻辑、性能影响、潜在风险和优化方案。
