# 每次变更完整工作流设计

每次代码或数据资产变更都通过 PR 主流程：开发生成、快速校验、完整校验、独立审查、修复、复核、人工审批和合并发布。特性分支允许生成与修复；`develop`、`master` 仅接受通过全部状态检查和 Code Owner 审批的 PR。

`scripts/test_hooks.py` 作为可重复执行的钩子自检，验证有效与无效提交信息，并验证本地钩子同步，不再依赖固定工作区路径。

GitHub Actions 负责可自动验证的阶段；GitHub 分支保护负责禁止直接推送、要求状态检查、Code Owner 审批和线性历史。

校验失败进入闭环：修复角色根据失败报告修改并重新执行完整校验，最多自动修复 3 次。任一次通过则进入独立审查与人工审批；连续 3 次失败时，工作流生成失败报告并创建人工审查任务，PR 保持不可合并、不可发布状态，等待人工决定修复方案或终止变更。
