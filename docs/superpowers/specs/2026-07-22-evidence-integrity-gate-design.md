# Harness 证据完整性门禁设计

## 目标

第 3 项整改用于防止任务使用不属于当前任务、结果未通过、内容被篡改或过期的证据推进状态。该整改只影响 Harness 门禁，不修改业务数据仓库资产。

## 设计

新增独立证据完整性校验模块，统一校验以下内容：

- `evidence_id` 非空、唯一，并与任务清单一致；
- `task_id` 必须等于当前任务编号；
- `purpose` 必须是当前阶段要求的证据类型；
- `result` 必须为 `passed`；
- `created_at` 必须是合法时间，不能晚于当前时间，并受证据有效期策略约束；
- `repository_revision` 必须存在；
- `file_read` 证据的路径必须位于仓库内、文件必须存在，且当前 SHA-256 必须与证据一致；
- 证据文件必须位于当前任务的 `evidence/` 目录内。

任务新增证据时立即执行结构校验，状态迁移前再次执行完整校验，避免只依赖正常写入入口。

## 错误处理

任何证据缺失、归属不符、结果非通过、时间非法、文件不存在或哈希不一致，都拒绝状态迁移，并输出具体证据编号和失败原因。不会自动修正或删除证据。

## 测试

覆盖有效证据、错误任务归属、非通过结果、缺失字段、文件哈希变化、文件路径越界、过期证据和重复证据编号；完成 Harness 单元测试及工作区完整校验。
