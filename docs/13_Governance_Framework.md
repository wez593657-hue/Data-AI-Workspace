# 13 治理框架

## 13.1 概述

本项目采用"五层约束"体系，将开发流程变成开发门禁，确保代码质量和开发规范的严格执行。

## 13.2 五层约束体系

```
┌─────────────────────────────────────────────────────────────┐
│                    第五层：人工审批                          │
│          (技术审批 → 业务审批 → 安全合规审批)                │
├─────────────────────────────────────────────────────────────┤
│                    第四层：自动校验                          │
│          (DDL校验 → 数据字典校验 → Mapping校验)             │
├─────────────────────────────────────────────────────────────┤
│                    第三层：检查清单                          │
│          (开发清单 → 审查清单 → 发布清单)                   │
├─────────────────────────────────────────────────────────────┤
│                    第二层：标准模板                          │
│          (需求模板 → 设计模板 → SQL模板)                    │
├─────────────────────────────────────────────────────────────┤
│                    第一层：仓库规则                          │
│          (CODEOWNERS → PR模板 → Issue模板)                 │
└─────────────────────────────────────────────────────────────┘
```

## 13.7 第五层：人工审批

### 13.7.1 审批流程

```
开发完成 → 自动校验通过 → 代码审查 → L1技术审批 → L2业务审批 → L3安全合规审批 → 合并发布
```

### 13.7.2 审批层级

| 层级 | 审批人 | 审批内容 |
|------|--------|----------|
| L1 | 技术负责人 | 技术方案、代码质量 |
| L2 | 业务负责人 | 业务需求、数据准确性 |
| L3 | 安全合规负责人 | 数据安全、合规性 |

### 13.7.3 审批标准

详细审批标准见审批流程文档。

```
文件: docs/15_Approval_Process.md
```

## 13.8 AI Agent 角色边界

### 13.8.1 开发Agent

**职责**:
- 代码实现
- 需求分析
- 设计文档编写

**禁止**:
- 执行校验操作
- 修改校验脚本
- 跳过审批流程

**操作范围**:
- `data_assets/ddl/`
- `data_assets/data_dictionary/`
- `data_assets/mapping/`
- `data_assets/etl/`
- `requirements/`

### 13.8.2 校验Agent

**职责**:
- 执行自动校验
- 生成校验报告
- 识别问题

**禁止**:
- 修改业务代码
- 修改DDL文件
- 修改数据字典

**操作范围**:
- 只读访问所有文件

### 13.8.3 审查Agent

**职责**:
- 代码审查
- 提出改进建议
- 验证修复

**禁止**:
- 直接修改业务代码
- 跳过审查步骤

**操作范围**:
- 只读访问所有文件

### 13.8.4 角色边界矩阵

| 操作 | 开发Agent | 校验Agent | 审查Agent |
|------|-----------|-----------|-----------|
| 修改DDL | ✓ | ✗ | ✗ |
| 修改数据字典 | ✓ | ✗ | ✗ |
| 修改Mapping | ✓ | ✗ | ✗ |
| 执行校验 | ✗ | ✓ | ✗ |
| 代码审查 | ✗ | ✗ | ✓ |
| 审批操作 | ✗ | ✗ | ✗ |

## 13.9 约束执行机制

### 13.9.1 门禁规则

- **PR门禁**: 自动校验未通过的PR不能合并
- **审查门禁**: 代码审查未通过的PR不能合并
- **审批门禁**: 人工审批未通过的PR不能合并

### 13.9.2 违规处理

| 违规类型 | 处理方式 |
|----------|----------|
| 跳过自动校验 | 拒绝合并 |
| 跳过代码审查 | 拒绝合并 |
| 跳过人工审批 | 拒绝合并 |
| 违反命名规范 | 返回修改 |
| 数据不一致 | 返回修改 |

### 13.9.3 责任追溯

所有变更记录完整的审计日志，可追溯到：
- 开发者
- 审查人
- 审批人
- 变更时间
- 变更内容

## 13.10 持续改进

### 13.10.1 定期评估

每季度对约束体系进行评估：
- 约束有效性
- 执行效率
- 开发者反馈

### 13.10.2 流程优化

根据评估结果优化：
- 约束规则
- 校验脚本
- 审批流程

### 13.10.3 工具升级

持续升级自动化工具：
- 校验工具
- 审查工具
- 监控工具
