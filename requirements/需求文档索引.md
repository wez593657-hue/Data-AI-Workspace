## 需求版本管理机制

### 1. 版本管理原则

| 原则 | 说明 |
|------|------|
| 唯一版本源 | 每个需求文档头部必须包含版本信息和变更记录 |
| 追溯性 | 每次需求变更必须记录变更原因、变更内容、变更人和审核状态 |
| 同步更新 | 需求变更后，相关的代码、数据字典、Mapping必须同步更新并标注需求版本 |
| 审核前置 | 需求变更必须经过评审和审核后，才能进入开发阶段 |

### 2. 版本号规则

- **主版本号**：重大业务变更（如新增模块、重大流程调整）
- **次版本号**：功能需求变更（如指标口径调整、逻辑变更）
- **修订号**：文档格式调整、文字修正、非逻辑变更

格式：`vX.Y.Z`

示例：`v2.1.0` 表示第二次重大版本下的第一次功能变更

### 3. 需求文档版本模板

每个需求文档头部必须包含以下信息：

```markdown
# 文档标题

## 版本信息

| 属性 | 值 |
|------|------|
| 版本 | vX.Y.Z |
| 创建日期 | YYYY-MM-DD |
| 最后更新 | YYYY-MM-DD |
| 状态 | 【待确认】/【已审核】/【开发中】/【已上线】 |

## 变更历史

| 版本 | 日期 | 变更内容 | 变更人 | 审核状态 | 关联需求编号 |
|------|------|----------|--------|----------|--------------|
| vX.Y.Z | YYYY-MM-DD | 变更描述 | 姓名/工号 | 已审核 | REQ-XXX |
```

### 4. 需求变更流程

```
用户提出修改 → 更新需求文档版本 → 更新代码标注 → 重新开发/审核
```

#### 流程说明

| 步骤 | 责任人 | 操作内容 | 输出物 |
|------|--------|----------|--------|
| 1. 用户提出修改 | 业务方/开发人员 | 提出需求变更内容和原因 | 变更内容描述 |
| 2. 更新需求文档版本 | 需求负责人 | 1. 更新需求文档正文内容<br>2. 更新版本号<br>3. 添加变更记录<br>4. 更新需求-代码映射 | 更新后的需求文档 |
| 3. 更新代码标注 | 开发人员 | 1. 更新代码头部需求版本号<br>2. 更新代码头部变更记录<br>3. 在变更逻辑处添加注释标记 | 更新后的代码文件 |
| 4. 重新开发/审核 | 开发人员/审核人员 | 1. 按更新后的需求重新开发<br>2. 按最新需求版本进行审核 | 审核通过的代码 |

### 5. 需求-代码映射

每个需求文档必须建立与代码的映射关系，在文档末尾增加：

```markdown
## 需求-代码映射

| 需求编号 | 需求描述 | 对应文件 | 状态 | 代码版本 |
|----------|----------|----------|------|----------|
| REQ-001 | 到期承接明细表 | PRC_ads_cust_deadline_rmnd_dtl.sql | 已实现 | v2.1 |
```

### 6. 审核前必查清单

| 检查项 | 检查内容 |
|--------|----------|
| 需求版本 | 确认需求文档版本号与代码标注一致 |
| 变更记录 | 检查是否有未审核的需求变更 |
| 映射关系 | 确认需求与代码的映射是否完整 |
| 影响范围 | 评估需求变更对其他模块的影响 |
