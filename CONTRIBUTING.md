# 贡献指南

## 代码托管

远程仓库地址: `https://github.com/wez593657-hue/Data-AI-Workspace.git`

## 钩子安装

克隆仓库后，必须安装Git钩子以启用本地门禁：

```bash
# 安装钩子
python scripts/install_hooks.py

# 验证钩子安装
ls -la .git/hooks/
```

钩子目录说明：
- `hooks/`: 钩子源文件（Git跟踪）
- `.git/hooks/`: Git钩子目录（本地，不跟踪）

安装的钩子：
- `pre-commit`: 提交前校验（数据资产一致性、文档审核、工作区检查）
- `pre-push`: 推送前检查（远程同步、提交信息格式、完整校验）
- `commit-msg`: 提交信息格式校验

## 分支说明（历史参考）

当前开发统一在 `master` 分支进行，以下命名仅保留用于理解历史提交，不再创建或使用。

| 分支类型 | 命名规范 | 用途 | 保护状态 |
|----------|----------|------|----------|
| 主分支 | `master` | 唯一开发和发布分支 | 允许直接推送 |
| 开发分支 | `develop` | 历史分支 | 不再使用 |
| 特性分支 | `feature/{feature-name}` | 历史分支 | 不再使用 |
| 修复分支 | `bugfix/{bug-name}` | 历史分支 | 不再使用 |
| 发布分支 | `release/{version}` | 历史分支 | 不再使用 |

## 开发流程

```
需求分析 → master开发 → 本地校验 → 用户确认脚本 → commit → push
```

### 详细步骤

| 步骤 | 操作 | 责任人 | 技术约束 |
|------|------|--------|----------|
| 1 | 需求分析 | 产品经理 | - |
| 2 | 进入主分支 | 开发人员 | `git checkout master` |
| 3 | 开发实现 | 开发人员 | 遵循编码规范 |
| 4 | 本地校验 | 开发人员 | ✅ Git钩子强制校验 |
| 5 | 用户确认脚本 | 用户 | 确认变更和校验结果 |
| 6 | 提交代码 | 开发人员 | `git commit` |
| 7 | 推送主分支 | 开发人员 | `git push origin master` |

## 提交规范

### 提交类型

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 新增客户风险评估表` |
| `fix` | Bug修复 | `fix: 修复DDL字段类型错误` |
| `docs` | 文档更新 | `docs: 更新数据字典规范` |
| `refactor` | 代码重构 | `refactor: 优化校验脚本` |
| `style` | 格式调整 | `style: 统一SQL缩进` |
| `test` | 测试相关 | `test: 添加单元测试` |
| `chore` | 构建/工具 | `chore: 更新依赖` |
| `sync` | 数据同步 | `sync: 同步Mapping文件变更` |

### 提交格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

## 门禁规则

### 第一层：仓库规则

- Issue模板: 规范问题描述

### 第二层：标准模板

- 需求文档模板
- 设计文档模板
- SQL模板
- 存储过程模板

### 第三层：检查清单

- 开发检查清单
- 审查检查清单
- 发布检查清单

### 第四层：自动校验

- DDL语法校验
- 数据字典一致性校验
- Mapping完整性校验
- SQL规范校验

### 第五层：用户确认

- 用户确认脚本内容、校验结果和风险后，允许直接提交并推送。

## 技术约束

### Git钩子

| 钩子 | 功能 | 校验内容 | 失败处理 |
|------|------|----------|----------|
| `pre-commit` | 提交前校验 | 数据资产一致性、文件生成校验、提交信息格式 | 阻止提交 |
| `pre-push` | 推送前检查 | 远程更新检查、校验状态 | 阻止推送 |

### CI流水线

| 流水线 | 触发条件 | 校验内容 | 失败处理 |
|--------|----------|----------|----------|
| `push-check.yml` | `master` 推送 | 数据资产校验、提交信息格式 | 流水线失败 |

### 主分支推送

- `master` 是唯一开发和发布分支，允许用户确认后直接推送。
- 不使用 PR、CODEOWNERS 或分支保护作为提交门禁。

## 流程约束

### AI Agent 使用规则

| Agent类型 | 职责 | 可读目录 | 可写目录 | 禁止操作 |
|-----------|------|----------|----------|----------|
| **开发Agent** | 代码实现、需求分析 | 所有目录 | `data_assets/`, `requirements/`, `docs/` | 执行校验脚本、修改校验脚本 |
| **校验Agent** | 质量检查、生成报告 | 所有目录 | 无 | 修改业务代码、提交代码 |

### 边界约束

| 约束类型 | 规则 | 违反处理 |
|----------|------|----------|
| 目录边界 | 开发Agent只能在指定目录内操作 | 人工干预 |
| 修改边界 | 校验/审查Agent只读，不得修改 | 人工干预 |
| 流程边界 | 必须按流程执行，不得跳过 | Git钩子/CI阻止 |

### 角色分离

```
开发Agent → 修改代码并执行校验
    ↓
用户 → 确认脚本、校验结果和风险
    ↓
开发Agent → commit 并 push 到 master
```

## 代码审查

### 当前流程

仓库不再设置 PR 代码审查流程。开发完成并通过必要校验后，由用户确认脚本，再直接提交和推送。

### 审查标准

- 代码规范性
- 逻辑正确性
- 安全性
- 可维护性

## 版本发布

### 发布流程

1. 在 `master` 完成开发
2. 执行发布检查清单
3. 测试验证
4. 用户确认脚本
5. 提交并推送到 `master`
